# 1C ↔ AutoParts B2B — Sync API Reference

This document describes the HTTP API used by the 1C side to keep the AutoParts B2B
catalog, stock, prices, categories, and customers in sync.

It is the single source of truth for the 1C developer.

---

## 1. Authentication

Every request to `/api/sync/1c/*` MUST include the header:

```
X-Sync-Token: <shared secret>
```

The server compares this value to `process.env.ONEC_SYNC_TOKEN` in
constant time. Missing or wrong token returns:

```
HTTP 401 { "ok": false, "error": "Invalid X-Sync-Token" }
```

### How to obtain the token

The token is a long random string generated by the AutoParts admin team. Treat
it like a password — do not commit it to source control or log it.
Rotation is supported by simply updating `ONEC_SYNC_TOKEN` on the server side
and notifying 1C of the new value.

---

## 2. Common response shape

All endpoints return:

```json
{
  "ok": true,
  "processed": 1234,
  "failed": 0,
  "syncLogId": "ckxx...",
  "status": "SUCCESS"
}
```

`status` is one of:

| Value     | Meaning                                                      |
|-----------|--------------------------------------------------------------|
| `SUCCESS` | every row processed, no failures                             |
| `PARTIAL` | some rows succeeded, some failed (see admin UI for details)  |
| `FAILED`  | fatal error or every row failed                              |
| `SKIPPED` | empty payload, no work done                                  |

`syncLogId` references a row in the `SyncLog` table — the admin UI at
`/admin/sync` shows the full history including any per-row error messages.

## 3. Endpoints

### 3.1 `POST /api/sync/1c/products`

Recommended frequency: **hourly**.

Full or delta upsert by `externalId`. The server is idempotent — sending the
same payload twice is safe.

**Body**

```json
{
  "mode": "delta",
  "products": [
    {
      "externalId": "1C-PRD-0001",
      "sku": "BRK-PAD-FRONT-001",
      "name": "Front Brake Pads — Toyota Camry",
      "description": "OEM-grade ceramic pads…",
      "brand": "Bosch",
      "categoryExternalId": "1C-CAT-BRAKES",
      "price": "84.50",
      "stock": 27,
      "isActive": true
    }
  ]
}
```

| Field                | Required | Notes                                          |
|----------------------|----------|------------------------------------------------|
| `mode`               | no       | `"delta"` (default) or `"full"`                |
| `products[]`         | yes      | up to 50 000 rows per request                  |
| `externalId`         | yes      | 1C primary key, ≤ 64 chars                     |
| `sku`                | yes      | unique across catalog                          |
| `name`               | yes      | ≤ 500 chars                                    |
| `description`        | no       | nullable                                       |
| `brand`              | no       | nullable                                       |
| `categoryExternalId` | no       | resolved against `Category.externalId`         |
| `price`              | yes      | string or number, e.g. `"84.50"` or `84.5`     |
| `stock`              | yes      | non-negative integer                           |
| `isActive`           | no       | defaults to `true` on create                   |

**`mode: "full"` semantics**

When `full`, AutoParts deactivates every sync-managed product whose
`externalId` is **not** in the payload (`isActive` is set to `false`).
Locally-created products (those with `externalId` starting with `local-`)
are never touched. Products are **never deleted** — order history must
remain intact.

**Example curl**

```bash
curl -X POST https://autoparts.example.com/api/sync/1c/products \
  -H "Content-Type: application/json" \
  -H "X-Sync-Token: $ONEC_SYNC_TOKEN" \
  -d '{
    "mode": "delta",
    "products": [
      { "externalId": "1C-PRD-0001", "sku": "BRK-PAD-001",
        "name": "Front Brake Pads", "price": "84.50", "stock": 27 }
    ]
  }'
```

### 3.2 `POST /api/sync/1c/stock`

Recommended frequency: **every 5–15 minutes**.

Lightweight stock-only refresh. Use this when only quantities change — much
cheaper than a full product sync.

**Body**

```json
{
  "items": [
    { "externalId": "1C-PRD-0001", "stock": 27 },
    { "externalId": "1C-PRD-0002", "stock": 0 }
  ]
}
```

Items whose `externalId` is unknown to AutoParts are counted as failures
with the message `"product not found — run /api/sync/1c/products first"`.

```bash
curl -X POST https://autoparts.example.com/api/sync/1c/stock \
  -H "Content-Type: application/json" \
  -H "X-Sync-Token: $ONEC_SYNC_TOKEN" \
  -d '{ "items": [{ "externalId": "1C-PRD-0001", "stock": 27 }] }'
```

---

### 3.3 `POST /api/sync/1c/prices`

Recommended frequency: **hourly**.

Same shape as stock, but for `price`.

```json
{
  "items": [
    { "externalId": "1C-PRD-0001", "price": "84.50" },
    { "externalId": "1C-PRD-0002", "price": 199 }
  ]
}
```

```bash
curl -X POST https://autoparts.example.com/api/sync/1c/prices \
  -H "Content-Type: application/json" \
  -H "X-Sync-Token: $ONEC_SYNC_TOKEN" \
  -d '{ "items": [{ "externalId": "1C-PRD-0001", "price": "84.50" }] }'
```

---

### 3.4 `POST /api/sync/1c/categories`

Recommended frequency: **daily**.

Two-pass upsert: AutoParts first creates/updates every category, then
resolves `parentExternalId → parentId`. Insertion order does not matter.

**Body**

```json
{
  "categories": [
    { "externalId": "1C-CAT-BRAKES",      "name": "Brakes" },
    { "externalId": "1C-CAT-BRAKE-PADS",  "name": "Brake pads",
      "parentExternalId": "1C-CAT-BRAKES" }
  ]
}
```

Cycles are detected and rejected — the offending row is recorded in
`errorDetails` with `"would create a cycle — parent ignored"` and saved
with `parentId: null`.

---

### 3.5 `POST /api/sync/1c/customers`

Recommended frequency: **daily** (or on-demand after balance changes).

```json
{
  "customers": [
    {
      "externalId": "1C-CST-001",
      "name": "Acme Auto Parts LLC",
      "taxId": "123456789",
      "phone": "+994 50 123 4567",
      "address": "Baku, Azerbaijan",
      "balance": "0.00",
      "creditLimit": "5000.00"
    }
  ]
}
```

`balance` is the customer's current outstanding amount on the AutoParts
account (typically negative when the customer owes money — but the API
accepts only non-negative values; use `0` if the account is clean).
`creditLimit` is the maximum debt allowed before orders are blocked.

## 4. Error handling and retries

| HTTP status | Meaning                                | Retry?                           |
|-------------|----------------------------------------|----------------------------------|
| `200`       | request accepted (check `status` field)| no, but inspect `failed` count   |
| `400`       | invalid JSON or schema validation      | no — fix the payload first       |
| `401`       | invalid `X-Sync-Token`                 | no — fix the token               |
| `5xx`       | server error                           | yes — exponential backoff        |

A `200 OK` with `"status": "PARTIAL"` is **not** a hard failure. The server
processed what it could and recorded per-row errors in `errorDetails` of the
SyncLog. The 1C side should log the `syncLogId` and continue with the next
batch. The AutoParts admin UI surfaces every PARTIAL/FAILED run for review.

### Idempotency

Every endpoint upserts by `externalId`. Re-sending the same payload is
**always safe** — failed batches can be retried verbatim.

### Recommended retry policy

* Network/timeout errors or HTTP 5xx — retry with exponential backoff
  (`30s`, `2min`, `10min`, then alert).
* HTTP 401/400 — do not retry, surface the error to a human.
* `status: "PARTIAL"` — log and continue.

## 5. Pull fallback (Vercel Cron)

In addition to the push endpoints, AutoParts can run as a pull client. Set
the following on the AutoParts server:

* `ONEC_PULL_URL` — base URL of the 1C HTTP service (e.g.
  `http://1c.example.com/autoparts/hs/exchange`)
* `ONEC_PULL_AUTH` — value of the `Authorization` header sent to 1C
  (e.g. `Bearer abc123` or `Basic dXNlcjpwYXNz`)

A Vercel Cron job runs every 30 minutes and issues `GET ${ONEC_PULL_URL}/{type}`
for each of `products`, `stock`, `prices`, `categories`, `customers`.
The 1C side must respond with the same JSON body shapes used by the push
endpoints (i.e. wrapped in `{ products: [...] }`, `{ items: [...] }`, etc).

If `ONEC_PULL_URL` is not configured the cron is a no-op — it logs a
SKIPPED SyncLog row per type and returns 200.

---

## 6. SyncLog schema

Every push, pull, or manual sync writes a row to `SyncLog`:

```
id                 String   (cuid)
source             String   "1C_PUSH" | "1C_PULL" | "MANUAL"
syncType           String   "products" | "stock" | "prices" | "categories" | "customers"
status             String   "SUCCESS" | "PARTIAL" | "FAILED" | "SKIPPED" | "RUNNING"
recordsProcessed   Int
recordsFailed      Int
errorDetails       Json?    array of { externalId?, error }
startedAt          DateTime
finishedAt         DateTime?
```

Visible at `/admin/sync` with click-to-expand error details.
